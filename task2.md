# 작업 지시서 — 자율신경맥파기(HRV) 검사 기능 신규 추가

## 배경
검사실 노트북에 연결된 HRV 기기가 결과를 이미지/PDF 파일로 자동 출력하며, 그 폴더는
구글드라이브와 동기화되어 있음. 기존 인바디/근력검사와 달리 이 검사는 기기가 이미
완성된 판정까지 포함한 결과지를 만들어내므로, 규림툴은 판정 계산 로직을 만들 필요 없이
① 핵심 수치 몇 개만 기록해 추이 추적, ② 원본 결과지 이미지 보존, ③ 그 수치+학술근거를
버무린 AI 해설 생성, ④ "환자와 함께보기" 화면에서 이미지+해설을 함께 노출 하는 역할만
담당함.

## 사전 확인 사항 (작업 시작 전 조사)
- 기존 구글폼 연동(13-3)에 쓰인 서비스계정(`secrets/google-service-account.json`) 및
  `@googleapis/sheets` 사용 방식 확인 — 이번엔 Sheets가 아닌 Drive API가 필요하므로
  가벼운 전용 패키지(`@googleapis/drive` 등, 구글폼 연동 때 googleapis 전체번들(203MB)
  문제로 sheets 전용 패키지로 교체했던 사례 참고해서 동일 원칙 적용)가 있는지 확인
- 원장님이 구글드라이브 해당 폴더를 이 서비스계정 이메일과 공유(읽기 권한)해주는 절차
  안내 문서로 별도 정리해서 전달할 것 (폴더 ID 확인 방법 포함)

## 대상 파일 / 신규 구조

### DB 스키마
- `HrvTestRecord` 신규:
  - patientId, testDate(자정 정규화 원칙 적용)
  - vascularHealthIndex (숫자), vascularHealthType (문자열, 기기 판정 그대로),
    avgPulse (숫자), stressIndex (숫자)
  - sourceImagePath (원본 결과지 파일 — 구글드라이브에서 다운로드해 로컬에 저장,
    기존 이미지 업로드 자동 리사이즈/압축 유틸 재사용)
  - aiCommentary (Text, AI 생성 해설)
  - measuredByStaffId, createdAt
- `ExamAcademicGuide` 신규 (또는 향후 다른 검사종류 확장 고려한 범용 테이블):
  - examType(현재는 "HRV" 고정값), content(원장 작성 학술 근거 텍스트, 자유 서술)
  - 향후 소변검사 등 추가 시 examType만 늘려서 재사용 가능하게 설계

### 화면/API
1. **결과지 가져오기 모달** (기존 구글폼 가져오기 UX 패턴 재사용)
   - "HRV 결과지 가져오기" 버튼 → 지정 구글드라이브 폴더의 최근 파일 목록 조회
     (파일명, 수정일, 가능하면 썸네일)
   - 파일명에서 환자명 추정 → 규림툴 환자 목록과 매칭해 기본 선택값으로 제시하되,
     직원이 직접 환자를 재검색/변경할 수 있게 함 (매칭 실패/동명이인 대비)
   - 파일 선택 시 이미지 미리보기를 화면에 크게 띄움
2. **수치 입력 폼**
   - 이미지 미리보기 옆/아래에 혈관건강지수/혈관건강도타입/평균맥박/스트레스지수
     입력 필드 (원장님이 결과지 보면서 그대로 옮겨 입력)
   - 저장 시: 이미지를 로컬에 다운로드+리사이즈 저장, HrvTestRecord 생성
3. **AI 해설 생성**
   - 저장 직후(또는 상세화면에서 버튼으로) AI 해설 생성 API 호출
   - 입력재료: 이번 회차 4개 수치 + 직전 기록 대비 변화량(있는 경우) +
     `ExamAcademicGuide`의 HRV 학술 근거 텍스트
   - 프롬프트 원칙(기존 academicHook/티칭지 원칙 그대로 적용):
     - 실제 수치를 뭉뚱그리지 않고 그대로 인용
     - 원장님이 입력한 학술 근거 내용만 재구성, 새로운 의학적 효능/통계 창작 금지,
       신중한 단서("~일 수 있습니다") 임의 확신형 전환 금지
     - 추이(개선/유지/악화)가 있으면 자연스럽게 언급
     - 자체검토(self-check) 지시문 포함
4. **환자와 함께보기 화면** (`/patient-view/exam/hrv/[id]` 신규)
   - 기존 검사 환자모드와 동일한 독립화면(사이드바 없음, 팝업 오픈)
   - 원본 결과지 이미지를 크게 표시 + 그 아래 AI 해설 텍스트
   - 수정/삭제 버튼 등 내부용 UI 노출 안 됨 (화이트리스트 방식 재사용)
5. **설정 화면**
   - `/settings/exam-guides` (또는 기존 설정 메뉴 내 새 탭) — 원장 전용, HRV 학술
     근거 텍스트 작성/수정
6. **환자 검사이력**
   - 기존 `/examinations/patient/[patientId]` 통합 타임라인에 HRV 회차도 함께 노출
     (썸네일 + 4개 수치 요약 + "환자와 함께보기" 바로가기)

## 검증 체크리스트
- [ ] 구글드라이브 폴더 파일 목록 정상 조회 확인

- [ ] 결과지 이미지 다운로드+리사이즈 저장 정상 확인 (대용량 파일 포함)

- [ ] 직전 기록 없는 최초 검사 시 "추이" 언급 없이 정상 생성되는지 확인
- [ ] "환자와 함께보기" 화면에서 내부정보 노출 없는지 확인

# 작업 지시서 추가사항 — "환자와 함께보기" 화면 이미지 확대/축소/드래그 기능

## 배경
HRV "환자와 함께보기" 화면(`/patient-view/exam/hrv/[id]`)에서 원본 결과지 이미지를
크게 띄우는데, 글씨가 작아 잘 안 보일 수 있어 확대/축소/드래그(팬) 기능 필요.

## 동작 규칙
- 이미지 위에서:
  - 마우스 휠 스크롤 → 확대/축소 (커서 위치 기준 확대되면 자연스러움)
  - 확대된 상태에서 드래그(마우스 클릭+이동) → 이미지 이동(팬)
  - 터치스크린/트랙패드 핀치 제스처도 지원 (모니터가 터치 지원 기기일 가능성 고려)
- 화면 한쪽에 작은 컨트롤 버튼(확대 +/축소 -/원래크기 복귀) 추가 — 마우스 조작이
  익숙치 않은 상황(환자 앞에서) 대비, 클릭만으로도 조작 가능하게
- 기본 진입 시에는 원래 크기(전체 이미지가 화면에 맞게)로 표시, 이후 사용자가
  확대/축소한 상태 유지 (같은 세션 내에서)
- 구현은 가벼운 방식 권장 — 별도 무거운 라이브러리 설치보다는 CSS transform(scale,
  translate) + 마우스/휠 이벤트 핸들링으로 직접 구현하거나, 이미 프로젝트에 적합한
  가벼운 라이브러리가 있다면 검토 (예: react-zoom-pan-pinch류 — 단, 신규 의존성
  추가이므로 필요성 판단해서 진행)
- 기존 화이트리스트 방식(내부정보 노출 차단)에는 영향 없어야 함 — 이미지 확대/축소는
  순수 뷰어 기능이므로 별도 데이터 노출 문제 없음

## 검증 체크리스트
- [ ] 마우스 휠로 확대/축소 정상 동작
- [ ] 확대 상태에서 드래그로 이미지 이동 정상 동작
- [ ] 버튼(+/-/원래크기) 클릭으로도 동일하게 조작 가능한지 확인
- [ ] 화면 진입 시 기본적으로 전체 이미지가 잘 보이는 크기로 시작하는지 확인
- [ ] 브라우저 확대/축소(Ctrl+휠 등 브라우저 자체 줌)와 충돌 없는지 확인
- [ ] npx tsc --noEmit, npm run build 통과
