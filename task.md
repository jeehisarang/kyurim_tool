# 작업 지시서 — 내원체크/환자관리 실사용 보완 (날짜이동 + 검증 + 수정/삭제)

## 배경 / 목적
실제 환자 데이터 입력이 시작되면서 아래 4가지 실사용 문제가 확인됨.
스키마 변경이 필요한 부분과 화면/검증 로직만으로 해결 가능한 부분이 섞여 있으므로,
작업 시작 전 현재 Patient/Visit 스키마 구조를 먼저 확인하고 진행할 것.

## 1. 내원체크 날짜 이동 기능 (`/visit-check`)
- 현재 `/todo`에 이미 구현된 날짜 이동 패턴(◀▶ + "오늘" 버튼 + ?date= 쿼리)을 동일하게 적용
- 상단에 날짜 이동 바 추가, 선택 날짜가 화면 제목("2026년 7월 6일 수요일")에 반영
- "환자 검색 → 신규/기존 체크" 흐름에서 저장되는 Visit의 visitDate가 **선택된 날짜 기준**으로
  저장되도록 변경 (오늘이 아닌 과거 날짜 선택 시 그 날짜로 기록)
- "오늘 체크된 내원 목록" → 선택 날짜 기준 목록 + 제목("7월 6일 체크된 내원 목록 (N건)")으로 전환
- 과거 날짜 선택 시에도 신규 환자 등록 및 체크가 정상 동작해야 함
- **저장 시 visitDate는 반드시 자정(00:00:00) 기준으로 정규화**해서 저장할 것
  (기존 트러블슈팅에서 확인된 `/api/visits` 자정 매칭 이슈 재발 방지 — 이번 기회에
  실사용 입력 경로에서도 자정 정규화가 일관되게 적용되는지 재점검)

## 2. 신규 환자 등록 — 입력 검증 + 중복 방지
### 2-1. 차트번호 형식 검증
- 차트번호 입력 필드에 검증 추가: 숫자만 허용 (한글/영문/특수문자 입력 시 저장 거부,
  안내 메시지 표시)
- 기존 데이터 중 이미 형식에 안 맞는 값이 있는지 확인 후 보고 (있어도 강제 수정은 하지 말고
  보고만 할 것 — 원장님 판단 필요)

### 2-2. 중복 환자 방지
- 신규 환자 등록 시 입력된 차트번호가 이미 존재하는 Patient와 일치하면:
  - 즉시 저장하지 말고 "이미 등록된 환자입니다: [이름] ([차트번호])" 안내와 함께
    "이 환자로 진행" / "다시 입력" 두 가지 선택지 제공
  - "이 환자로 진행" 선택 시 새 Patient를 생성하지 않고 기존 환자로 체크 진행
- 이름만 같고 차트번호가 다른 경우(동명이인 가능성)는 별도 경고 없이 정상 진행
  (차트번호가 유일 식별자이므로)

## 3. 환자 정보 수정 기능
- 환자 검색 결과 또는 상세 화면에서 이름/차트번호를 수정할 수 있는 UI 추가
  (신규 등록 화면과 별도로, 기존 환자 검색 후 "정보 수정" 버튼 또는 인라인 편집 방식 —
  기존 화면 패턴에 맞춰 판단)
- 차트번호 수정 시에도 위 2-1 검증(숫자만) 및 2-2 중복 체크(다른 환자와 충돌 여부) 동일 적용
- 수정 이력을 별도로 남길 필요는 없음 (단순 UPDATE, "가볍고 쉬운 구조" 원칙 유지)

## 4. 잘못된 내원 체크 기록 수정/삭제
- `/visit-check`의 "체크된 내원 목록"에서 각 항목에 수정/삭제 버튼 추가
- 수정: 진료분야/진료구분/예약여부 변경 가능
- 삭제: 기존 원칙(`isActive` 비활성화 방식, 물리적 삭제 아님)을 Visit 레코드에도 동일 적용
  — Patient와 마찬가지로 통계 정확성을 위해 완전 삭제가 아닌 비활성화 처리
- 비활성화된 Visit은 통계/목록/카운트에서 전부 제외되는지 확인 (기존 Patient.isActive
  처리 로직과 동일한 방식으로 필터링)

## 5. 회귀 테스트
- 날짜 이동 후 과거 날짜로 신규 환자 등록 + 체크 → 해당 날짜 목록에 정상 반영되는지 확인
- 차트번호에 문자 입력 시 저장 거부되는지 확인
- 기존 차트번호로 재등록 시도 시 중복 안내 뜨는지, "이 환자로 진행" 선택 시 Patient가
  중복 생성되지 않는지 확인
- 환자 정보 수정 후 통계(누적방문수 등)에 정상 반영되는지 확인
- Visit 삭제(비활성화) 후 해당 건이 통계/목록에서 제외되는지 확인
- npm run build 통과 확인
