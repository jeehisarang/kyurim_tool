# 작업지시서 — 유비오맥파 CSV 자동연동 (HRV 자동 임포트) 설계

## 1. 배경

유비오맥파(Pro) HRV 기기가 원장실 PC에 설치되어 있고, 기기 소프트웨어의
"엑셀 변환" 기능으로 측정 데이터가 CSV로 누적 저장됨. 저장 경로는 Google
Drive 데스크탑 앱으로 동기화되는 로컬 폴더(예: G:\내 드라이브\...)로
지정돼 있어, 규림툴 서버 PC에서 로컬 파일시스템 경로로 직접 접근 가능.

목표: 직원이 HRV 수치를 수동으로 입력하지 않아도, 이 CSV를 규림툴이 주기적
으로 읽어서 HRV 검사기록을 자동 생성하도록 한다. 기존 수동 입력 화면(이미지
업로드+수치입력)은 자동연동 실패 시 폴백 수단으로 계속 남겨둔다 (삭제하지
말 것).

## 2. 실측으로 확인된 CSV 구조 (원장님이 실제 파일 확인 완료 — 아래는 확정된 사실)

- 인코딩: CP949(EUC-KR). UTF-8로 그대로 읽으면 한글이 깨짐 —
  iconv-lite 등으로 CP949→UTF-8 디코딩 필수
- 파일 위치: Google Drive 동기화 폴더 하위 맥파기검사기록지/csv/ 안에
  날짜별 파일(20260716.csv 형태)로 보임. 즉 "엑셀 변환" 버튼을 누른 날짜
  기준으로 새 파일이 생기고, 같은 날 여러 번 누르면 같은 파일에 계속
  누적되는 구조로 추정됨 — 실제로 다음날 파일이 새로 생기는지는 아직
  미확인이므로, 구현 시 "파일이 여러 개 존재할 수 있다"는 전제로 설계할 것
  (폴더 내 모든 csv 파일을 스캔하는 방식 권장, 특정 파일명 하드코딩 금지)
- 컬럼 순서 (헤더 그대로):
  사용자명, 성별, 출생년도, 나이, 번호, 측정일시, 수신감도, 혈관건강지수,
  혈관건강도타입, a-b(ms), a-c(ms), a-d(ms), a-e(ms), b/a, c/a, d/a, e/a,
  평균맥박, 스트레스지수, 상대스트레스, 복잡도, TP(Ln), VLF(Ln), LF(Ln),
  HF(Ln), LF/HF, SDNN, RMSSD
- 중요한 함정 — 측정 모드에 따라 컬럼이 부분적으로 비어있음: "혈관건강도
  측정"(1분, 단독)만 하면 혈관건강지수/혈관건강도타입/평균맥박 정도까지만
  채워지고, 스트레스지수·TP·VLF·LF·HF·LF/HF·SDNN·RMSSD는 빈 값으로 나옴.
  "스트레스 지수 측정"(2.5~5분)까지 진행해야 전체 컬럼이 채워짐. 실측
  샘플에서 실제로 이 패턴 확인됨(같은 환자의 동일 세션 내 첫 행은 부분만,
  둘째 행은 전체 채워짐)
- 번호(차트번호) 컬럼: 필수 입력이 아니라서 비어있는 행이 실제로 존재함
  (실측 샘플에서 확인)

## 3. 착수 전 확인 필요 (설계 먼저 제시)

구현 전 아래를 코드/환경에서 확인하고, 계획을 정리해서 먼저 보고할 것
(승인 후 실제 구현 진행):

1. 폴더 접근 확인: 규림툴 서버(원장실 PC)에서 Google Drive 동기화 폴더
   경로(G:\내 드라이브\규림한의원천안점(공유용)\맥파기검사기록지\csv\
   등, 원장님께 정확한 절대경로 재확인)에 실제로 접근 가능한지 확인
2. 이미지 캡처 파일 연동 가능 여부: 맥파기검사기록지 폴더 하위
   capture 폴더에 실제로 사람이 보는 결과화면 캡처(jpg/png류)가 쌓이는지,
   아니면 .ppg/.adc/.hrv 같은 원시 파형 데이터 파일만 있는지 확인.
   (스크린샷 상으로는 사용자명+측정일시가 파일명에 들어간 원시데이터
   파일들만 확인됐고, capture 폴더 내부는 아직 확인 안 됨) — 원시데이터
   파일은 이미지가 아니므로 이번 자동연동 범위에서 제외하고, 사람이 보는
   캡처 이미지가 별도로 있다면 그것만 파일명(측정일시 기준)으로 매칭해
   함께 저장하는 방안 검토
3. 기존 HRV 등록 로직과의 통합 지점 확인: 현재 수동 등록(이미지+수치
   입력) 시 어떤 함수/API로 HrvTestRecord가 생성되는지 확인해서, 자동
   임포트도 동일한 생성 로직(검증, AI 코멘트 생성 트리거 등)을 재사용할 수
   있는지 확인

## 4. 설계 방향 (원장님 결정사항 반영)

1. 상세지표도 함께 저장: 지금 화면에 노출하는 4개 지표(혈관건강지수,
   혈관건강도타입, 스트레스지수, 평균맥박) 외에, TP/VLF/LF/HF/LF-HF비율/
   SDNN/RMSSD도 함께 파싱해서 저장한다. 아직 UI에 노출하지 않아도 되며,
   향후 "HRV 학술근거 확장" 작업 때 재파싱 없이 바로 쓸 수 있도록 미리
   확보해두는 목적. (스키마에 각각 개별 컬럼으로 넣을지, JSON 필드 하나에
   묶어 넣을지는 Claude Code가 기존 스키마 패턴에 맞춰 제안할 것)

2. 부분 데이터 처리: "혈관건강도 측정"만 한 행(상세지표 없음)도 그냥
   버리지 말고, 있는 값만 저장하고 없는 값은 null로 둔다. 다만 이 경우
   원장님/직원에게 "스트레스 지수 측정까지 완료해야 상세 코멘트가 나온다"는
   안내가 필요 — 자동 임포트 결과 화면에 "일부 지표 누락" 표시를 하는 방안
   제안

3. 차트번호 매칭 로직:
   - CSV의 "번호" 필드가 채워져 있고 실제 환자 차트번호와 일치하면 → 자동으로
     해당 환자의 HrvTestRecord로 생성
   - "번호"가 비어있거나 매칭되는 환자가 없으면 → 자동 생성하지 않고 "미매칭
     검사기록" 대기열에 보관, 직원이 나중에 화면에서 환자를 수동 지정하면
     그때 정식 HrvTestRecord로 전환되는 구조 제안 (이 대기열 UI가 신규로
     필요함 — 위치는 기존 "치료처방"의 "오늘 할 일" 목록과 유사한 자리에
     배치하는 안 검토)

4. 중복 방지: 같은 CSV를 여러 번 스캔해도 같은 측정 기록이 중복
   생성되지 않도록, (사용자명 + 측정일시) 조합을 유니크 키로 판단해 이미
   처리한 행은 건너뛰는 방식 제안. 마지막으로 처리한 시점/행을 기록해두는
   방법(파일별 처리 상태 테이블)도 함께 설계할 것

5. 폴링 방식: 별도 API 인증 없이 로컬 파일 접근이 가능하므로, 파일 변경
   감지(예: chokidar) 또는 주기적 스캔(예: 5~10분 간격) 중 어느 쪽이 이
   환경에 더 적합한지 Claude Code가 판단해서 제안. 서버 재시작 시에도
   기존 처리 이력이 남아있어야 함(중복 재처리 방지)

6. 기존 수동 등록 화면 유지: 자동 임포트가 기본 경로가 되더라도, 자동
   연동이 실패하거나 매칭이 안 되는 경우를 위해 기존 수동 등록 화면은
   그대로 남겨둔다.

## 5. 이번 작업에서 하지 말아야 할 것

- 실제 자동 임포트 기능을 바로 구현하지 말 것 — 이번엔 위 1~5 확인 후
  설계안(스키마 변경안, 매칭/대기열 UI 초안 포함)까지만 정리해서 먼저
  보고할 것. 구현은 설계 승인 후 별도 지시로 진행
- 기존 HRV 수동 등록 화면 로직 변경 금지
- 실 환자 데이터 대상 스크립트 실행 금지 — 조사/설계 단계에서는 파일 구조
  확인만 하고, 실제 파싱 테스트가 필요하면 테스트 환자(426)로 진행

## 6. 완료보고 형식

- 3번(착수 전 확인) 결과 + 4번의 설계안(스키마 변경안, 매칭 로직, 미매칭
  대기열 UI 초안)을 정리해서 보고
- 특히 이미지 캡처 폴더 실제 내용, 파일이 날짜별로 여러 개 생기는지 여부는
  반드시 확인해서 보고할 것
