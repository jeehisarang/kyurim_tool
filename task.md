## 작업 지시서 — 프로그램티칭(ProgramTeaching) 이미지 업로드 시 자동 리사이즈/압축

### 배경
- 원장실에서 Claude Design으로 만든 보폐맥 프로그램티칭 이미지(5.15MB, PNG)를 
  `/settings/program-teaching`에서 업로드 시도 → "서버에 연결하지 못했습니다" 
  에러로 저장 실패
- 원인 추정: Next.js API route의 요청 본문 크기 제한(기본값이 낮게 설정된 경우가 
  많음)을 초과하면서 명확한 에러 메시지 대신 네트워크 에러처럼 보이는 것으로 판단됨
- 앞으로 나머지 9개 프로그램에도 유사한 고화질 이미지를 계속 업로드할 예정이므로, 
  매번 원장님이 파일 크기를 수동으로 줄여야 하는 번거로움을 없애고, 환자가 
  `/s/[token]`, `/p/[token]`에서 이미지를 볼 때 로딩 속도도 개선할 겸 근본 해결할 것

### 대상 파일 (Claude Code가 정확한 경로 재확인)
- 프로그램티칭 이미지 업로드 API 라우트 (`/api/program-teaching` 또는 관련 
  이미지 업로드 엔드포인트)
- 7/10 세션에서 만들어둔 "이미지 리사이즈 유틸" 재사용 가능 여부 확인 (당시 
  티칭지 관련 기능에 이미 구현되어 있었다고 기록됨 — 있으면 그대로 재사용, 
  없거나 범위가 다르면 신규 작성)
- Next.js route handler의 요청 본문 크기 제한 설정 (route segment config 또는 
  next.config.ts)

### 동작 규칙
1. **업로드 허용 크기 상향**: 원본 그대로 받을 수 있도록 요청 크기 제한을 
   최소 10~15MB 수준으로 올릴 것 (원장님이 원본을 그대로 올려도 실패하지 않도록)
2. **서버단 자동 리사이즈**: 업로드된 이미지를 저장하기 전에 
   - 가로/세로 중 긴 변 기준 **최대 1080px**로 리사이즈 (그 이상 해상도는 
     화면에서 어차피 의미 없음)
   - 품질 80% 내외로 압축 저장 (JPEG 또는 WebP — 기존 프로젝트에서 쓰던 
     포맷 컨벤션 확인 후 통일)
   - 목표: 결과 파일이 대략 200KB~500KB 수준이 되도록
3. 리사이즈/압축은 **서버에서 처리** (클라이언트단 처리 금지 — 원장님이 
   그림판으로 미리 줄여올 필요 없게, 원본 그대로 올려도 되게)
4. 리사이즈 실패 시(손상된 파일 등) 명확한 에러 메시지 반환 — 지금처럼 
   "서버에 연결하지 못했습니다" 같은 모호한 문구 대신 "이미지 처리 중 
   문제가 발생했습니다" 등 원인을 알 수 있는 메시지로 개선
5. 이미 저장된 기존 이미지(있다면)는 이번 작업 대상 아님 — 신규 업로드분부터 적용

### 검증 체크리스트
- [ ] 실제 5.15MB 원본 파일(보폐맥 이미지)로 업로드 시도 → 정상 저장 확인
- [ ] 저장된 파일의 실제 용량이 500KB 이하로 줄어들었는지 확인
- [ ] 리사이즈 후에도 육안상 화질 저하가 거슬리지 않는 수준인지 확인 
      (텍스트 가독성 포함)
- [ ] `/s/[token]`, `/p/[token]` 공개 페이지에서 로딩 속도 체감 개선 확인
- [ ] 작은 용량(500KB 이하) 이미지 업로드 시 기존처럼 정상 동작(회귀 없음) 확인
- [ ] 손상된 파일/비이미지 파일 업로드 시 명확한 에러 메시지 확인
- [ ] npx tsc --noEmit, npm run build 통과
