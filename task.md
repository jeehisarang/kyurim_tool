[작업지시서] 변증 체크리스트 기능 — 설계 조사 및 스키마 제안 (구현 아님, 1단계)

## 배경
- 현재 "/settings/exam-guides"의 "한의학적 가능성 매핑표"는 변증명 + 증상 키워드로 
  구성되어 있고, AI가 환자 증상기록(자유 텍스트) 안에서 이 키워드와 일치하는 경우에만 
  변증을 언급하는 방식.
- 오늘 확정한 새 방향: 이 키워드들을 환자가 직접 체크하는 "예/아니오 질문" 형태로 변환해, 
  환자 자가 응답 기반으로 변증 후보를 산출하고, 각 변증에 "치료원칙"(예: 간기울결→소간이기)을 
  추가로 매핑해 AI 코멘트에 구체적 치료 방향(한약/침 치료)까지 언급하도록 확장할 예정.
- 재현성이 낮은 기기 수치(자율신경균형도/맥박다양성) 기반 판정과 달리, 이 체크리스트는 
  안정적인 근거이므로 좀 더 적극적인 치료 유도 문구를 써도 된다는 원장 판단 있음.
- 사용 방식: (1) 원장실에서 검사와 함께 그 자리에서 태블릿 등으로 체크, 
  (2) 공유링크(/s/[token])로 환자가 나중에 직접 체크 — 두 경로 모두 지원 필요.
- 체크리스트 결과에 따른 치료 추천은 원장 확인 절차 없이 바로 환자 공개 화면에 반영됨 
  (원장 승인).

## 이번 단계에서 할 일 (구현 금지, 조사·설계 제안만)
1. 현재 "한의학적 가능성 매핑표" 관련 실제 Prisma 스키마(테이블명, 필드명, 관계) 확인
2. 기존 증상 키워드 매칭 로직(코드 경로) 확인 — AI가 어떻게 환자기록 텍스트와 
   키워드를 비교하는지
3. 아래 요구사항을 만족하는 스키마 확장안 설계 (실제 코드 변경 없이 제안만):
   - 변증별 "치료원칙" 필드 추가 (기존 매핑표 테이블에 컬럼 추가 또는 관계 테이블 신규 생성)
   - 변증별 체크리스트 질문 (기존 키워드를 질문형으로 변환한 문항 리스트, 순서 포함)
   - 환자 응답 저장 구조 (환자ID, 검사ID 또는 독립 응답, 체크된 문항들, 계산된 변증별 점수, 
     최종 후보 1~2개)
   - 원장실 입력용 UI 진입점 + 공유링크(/s/) 입력용 UI 진입점, 두 경로가 같은 저장 구조를 
     쓸 수 있는지 여부
4. 위 설계안을 문서(schema-proposal.md 등)로 정리해 원장 확인 요청 — 이번 단계에서는 
   실제 마이그레이션이나 코드 구현 하지 말 것

## 완료 보고 형식

- 기존 키워드 매칭 로직 요약
- 제안 스키마 (신규/변경 필드, 테이블 관계 다이어그램 또는 텍스트 설명)
- 원장실 입력 UI와 공유링크 입력 UI가 데이터 구조를 공유할 수 있는지에 대한 의견
- 궁금한 점이나 애매한 설계 결정 사항 (있다면 반드시 여기 나열, 임의로 결정하지 말 것)
