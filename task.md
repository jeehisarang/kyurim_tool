# 작업지시서 — 건강 리포트: 환자용/원장용 분리 + 미병 프레임 복원 + 클로징 헤드라인

## 배경
- 카드7(치료방향)에 지금까지 소간해울·보기익기 등 변증명+한자가 그대로 노출되고 있음.
  원장(윤우영)이 "환자에게 보여지는 화면은 한의학 전문용어 없이 쉬운 말로"를 요청.
- 미병(未病) 개념이 초기 설계(7/17) 때는 코멘트 철학의 중심이었으나, 이후 여러 라운드를
  거치며(교과서적 피드백→롤백, 근거설명 문장 금지 등) 카드7에서는 사실상 사라짐.
  카드4에는 "여러 미병 신호가 함께 나타나는 경향" 문구로 이미 남아있음 — 이건 그대로 유지.
- 카드1(헤드라인)과 카드7 마지막에 미병/치미병 개념을 다시 살려서, 리포트 전체를 열고
  닫는 구조로 재구성하기로 함.
- 카드7 끝에 그날 체크된 후보 카테고리 전체를 하나로 묶어 각인시키는 "클로징 헤드라인"을
  신규로 추가하기로 함 (기존 카드1 헤드라인과는 별개 — 위치도 역할도 다름, 혼동 주의).

## 원칙 (반드시 지킬 것)
- **원장용 카드7은 그대로 유지한다.** 변증명(소간해울 등) + 한자 병기 + 방제 계열, 기존
  `applyDeterministicBold` 로직 그대로. 이건 원장이 상담 중 참고하는 용도로 계속 존재.
- **환자용 카드7을 신규로 추가한다.** 같은 카테고리·같은 치료원칙을 근거로 하되, 변증명·
  한자·방제명은 전부 빼고 쉬운 말로 AI가 재구성. 두 버전은 같은 `treatmentPrinciple`을
  입력으로 삼아 AI가 동시에 생성(별도 호출 2번이 아니라 한 번의 호출에서 두 필드로
  출력하는 방식 권장 — 토큰/비용 고려해 Claude Code가 판단).
- **카드1/카드4/카드5의 기존 문구 로직은 건드리지 않는다.** 카드1에만 미병 언급을 자연스럽게
  얹는 프롬프트 보강만 추가.
- **카드5/6 관련 정책, redFlagNotice 조건부 노출, 안전장치(violatesZoneRestatementRule,
  violatesDuplicateSentenceRule, ensureArrhythmiaNotice 등) 전부 변경 없음.**
- **환자용 콘텐츠는 카드7뿐 아니라 검사톡 공유링크(`/s/[token]`, `/p/[token]`)에도 동일하게
  적용된다.** 환자가 보는 화면은 항상 환자용 버전만 노출, 원장용은 원장 확인화면
  (`/patients/[id]/health-report` 등 내부 화면)에서만 노출.

## 1. 스키마 변경
- `TcmCategory`에 필드 추가: `plainTreatmentText`(nullable, String) — 환자용 치료방향
  AI 생성 결과 저장용. 기존 `treatmentPrinciple`(원장용 입력 근거) 필드는 그대로 유지.
- 리포트 레코드(`HealthReportV1` 등 실제 테이블명 확인 후) 저장 스키마에 `closingHeadline`
  (nullable, String) 필드 추가 — 클로징 헤드라인 AI 생성 결과 저장용.
- 레거시 호환: 기존 저장된 리포트는 두 필드 모두 null. 렌더링 시 null이면 환자용 카드7/
  클로징 섹션 자체를 노출하지 않음(에러 아님, 조건부 렌더링) — "리포트 다시 만들기"
  실행 전까지는 기존 화면 그대로 유지.

## 2. AI 프롬프트 변경

### 2-1. 카드1(헤드라인) 프롬프트 보강
- 기존 "인정형+완만한경과" 톤 유지, 여기에 미병(未病) 개념을 자연스럽게 1회 언급하도록
  지시문 추가. 예시 톤:
  > "몸이 아직 병은 아니지만 보내는 신호 — 미병(未病) — 인 [패턴명] 경향을 살펴볼
  > 필요가 있습니다."
- 강제 삽입이 아니라 "자연스럽게 녹일 것" 지시 + 자체검토 항목에 "미병 언급이 문장
  흐름을 어색하게 만들지 않는지 확인" 추가.

### 2-2. 카드7 — 원장용/환자용 동시 생성
- 카테고리별 AI 호출 시, 출력 포맷을 2개 필드로 분리 요청:
  - `doctorText`: 기존과 동일 (변증명+한자+방제 계열, `applyDeterministicBold` 대상)
  - `patientText`: 환자용, 아래 원칙 적용
    1. 변증명·한자·방제명 전부 금지
    2. 문장 구조: "[증상 상태 설명]. [치료 방향]을 고려할 수 있습니다."
    3. 환자가 실제 체크한 해당 카테고리 증상 키워드를 문장 안에 자연스럽게 포함,
       **볼드 처리**(`<b>` 또는 마크다운 `**`, 기존 렌더러 방식 확인 후 통일)
    4. 치료 방향을 설명하는 핵심 어구("00한 치료")도 카테고리 색상에 맞춰 **볼드 처리**
    5. "구체적인 치료는 상담을 통해 결정됩니다" 류 공통 문구는 카테고리별 텍스트에
       넣지 말 것 — 카드7 전체 하단에 한 번만 공통 캡션으로 렌더링(컴포넌트에서 처리,
       AI 출력에 포함 안 함)
  - 자체검토 항목 추가: "patientText에 한의학 전문용어(변증명·방제명·한자)가 하나라도
    남아있으면 안 됨" — 위반 시 재시도, 최종 실패 시 기존 안전장치와 동일 원칙(코멘트
    실패 처리 또는 doctorText 기반 폴백은 Claude Code 판단)

### 2-3. 클로징 헤드라인 — 신규 AI 생성
- 입력: 그 환자의 후보 카테고리 전체 목록(1~4개, 동점 포함)
- 출력: 후보 카테고리들을 "따로가 아니라 함께 나타난다"는 흐름으로 엮은 1~2문장 +
  미병(未病)·치미병(治未病) 정의를 짧게 포함한 마무리 문장
- 카테고리 1개일 경우엔 "여러 신호가 함께" 같은 복수 표현을 쓰지 않도록 조건 분기
  (자연스러운 단수 표현으로 대체)
- 예시 톤 (최승희님 4개 카테고리 기준):
  > "긴장, 기력저하, 순환저하가 따로가 아니라 함께 나타나는 지금, 미병(未病)—아직
  > 병은 아니지만 균형이 무너지기 시작한 상태의 신호입니다. 병이 되기 전에 미리
  > 다스리는 치미병(治未病)이, 지금 필요한 치료입니다."
- 자체검토 항목: "미병/치미병 정의가 왜곡 없이 정확한지", "과도한 불안 조성 표현이
  없는지"(기존 카드5 원칙과 동일 기준 재사용)

## 3. UI/컴포넌트 변경

### 3-1. 카드7 렌더링
- 원장 확인화면: 기존 그대로(`doctorText` 렌더링, 변경 없음)
- 환자용 화면(환자와 함께보기 팝업 + `/s/[token]` + `/p/[token]`): `patientText` +
  카테고리 컬러 아이콘 카드 렌더링. 볼드 구절은 카테고리 색상(TCM_CATEGORY_COLOR_MAP
  참조, 800 단계 진한 톤)으로 강조.
- 카드7 최하단에 공통 캡션 1줄(작은 글씨, `text-muted`): "현재 증상에 맞는 구체적인
  치료는 상담을 통해 결정됩니다" — 컴포넌트 레벨 고정 텍스트, AI 생성 아님.

### 3-2. 클로징 헤드라인 카드
- 흰 배경(`surface-2`) + 얇은 테두리, 검은 글씨(`text-primary`) 기본
- "미병(未病)", "치미병(治未病)", 그리고 "따로가 아니라 함께 나타나는" 등 핵심 구절만
  볼드 + accent 컬러로 강조 (전체 볼드/전체 컬러 금지 — 대비를 위해 부분 강조만)
- 폰트 크기는 본문(14px)보다 한 단계 큼직하게(17~18px 선), 카드7 맨 아래, 공통 캡션
  문구 바로 위에 위치

### 3-3. 원장/환자 뷰 토글 (필요 시)
- 원장 확인화면에 "환자용 미리보기" 토글 버튼 추가하면 좋음 — 원장이 발송 전 환자용
  텍스트를 미리 확인 가능하게. 우선순위 낮음, 시간 되면 포함.

## 4. 검증 체크리스트
- 최승희(실제 저장 데이터, 후보 4개) 기준: doctorText/patientText/closingHeadline 3종
  모두 정상 생성 확인
- 후보 1개/2개/3개 케이스(테스트환자 426)로 클로징 헤드라인 단수/복수 표현 분기 확인
- patientText에 변증명·한자·방제명이 하나도 안 남는지 5회 반복 생성으로 확인
  (기존 볼드 적용률 검증과 동일 방식)
- 레거시 리포트(plainTreatmentText/closingHeadline null) 렌더링 시 에러 없이 해당
  섹션만 조건부로 안 뜨는지 확인
- `/s/[token]`, `/p/[token]` 양쪽에서 환자용 텍스트만 노출되고 원장용 텍스트는 절대
  섞이지 않는지 확인
- npx tsc --noEmit / npm run build 통과
- 완료 후 커밋만 하고 push는 보류 — 원장 브라우저 직접 확인 후 push 승인 받을 것
