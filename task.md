## 작업 지시서 — 치료처방 회차 계산 버그 수정 + 회차별 날짜 개별 수정 기능

### 배경
- 송호철님 환약·3개월 사례로 발견: 등록일 2026.7.13, 총기간 90일(3개월), 
  2주 간격 SPLIT 방식인데, 회차별 스케줄이 1차 7/27부터 시작해서 7차가 
  10/19에 끝남 — 90일(=10/11) 기한을 8일이나 초과함
- 추정 원인: 등록일 자체에 실제로 첫 회차 약을 받았을 텐데, 이걸 "1차"로 
  안 치고 **14일 뒤를 1차로 계산**하면서 전체가 한 텀씩 밀린 것으로 보임 
  (요구사항 문서 기준 "3달=총 6~7차"는 등록일을 1차로 포함해야 90일 안에 
  맞아떨어짐)
- 추가로, 실사용 중 환자가 정확히 2주 간격으로 안 오는 경우가 흔해서, 
  회차 예정일을 상세페이지에서 개별 수정할 수 있어야 함 — 단, **수정한 
  회차 하나만 바뀌고 나머지 회차는 원래 계산대로 유지** (연쇄 밀림 없음)

### 1단계 — 회차 계산 버그 조사 및 수정
- `computeSplitSchedule()` 등 회차 계산 로직 파일을 먼저 확인해서, 등록일이 
  1차로 포함되는지 아닌지 정확히 진단할 것 (제 추정이 틀렸을 수 있으니 
  실제 코드 확인 우선)
- 확인 후, **등록일 = 1차(이미 처방 완료)**로 간주하도록 수정
  - 1차: 예정일=등록일, 완료여부=완료, 완료일=등록일 (자동 완료 처리)
  - 2차부터 등록일 기준 14일, 28일... 간격으로 계산, 총회차(90일÷14일 
    반올림 등 기존 계산 방식) 안에서 마지막 회차가 등록일+90일을 넘지 
    않도록 확인
- **기존에 이미 등록된 다른 환자들의 치료처방(탕약/환약/킬팻1·3개월)도 
  동일 버그의 영향을 받았을 가능성이 높음** — 전체 진행중인 처방 목록을 
  조회해서 영향 범위 파악 후, 원장님께 몇 건이 영향받는지 먼저 보고할 것 
  (일괄 재계산 전 확인 필요)
- "오늘 할 일"에 이미 생성된 TodoTask(NEXT_DOSE)와의 정합성도 같이 확인 — 
  회차 계산 수정으로 인해 이미 생성된 할일과 날짜가 어긋나지 않는지 점검

### 2단계 — 회차별 날짜 개별 수정 기능
- 스키마: `Prescription`에 회차별 수동 조정값을 저장할 구조 추가 
  (예: 신규 테이블 `PrescriptionRoundOverride`(prescriptionId, roundNumber, 
  overrideDate) — roundNumber당 1건, 있으면 계산값 대신 이 값을 사용)
- 상세페이지(`/prescriptions/[prescriptionId]`) 회차별 스케줄 테이블에 각 
  행마다 "날짜 수정" 버튼/인라인 편집 추가
- API: `PATCH /api/prescriptions/[prescriptionId]/rounds/[roundNumber]` — 
  overrideDate 저장
- **수정한 회차만 예정일이 바뀌고, 그 앞뒤 다른 회차들의 예정일은 원래 
  계산값 그대로 유지** (연쇄 재계산 없음 — 확정)
- 수정된 회차와 연결된 TodoTask(있다면)의 예정일도 함께 갱신 (오늘 할 일 
  화면과 상세페이지가 어긋나지 않도록)
- 완료 처리된 회차(완료일 있는 경우)는 날짜 수정 불가 처리 (이미 지나간 
  기록은 건드리지 않는 기존 원칙 유지) — 수정은 "예정" 상태인 회차에만 허용
- 원복(초기화) 기능도 추가: 수동 수정한 회차를 다시 "계산값으로 되돌리기" 
  버튼
