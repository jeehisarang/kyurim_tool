# 작업 지시서 — 신청폼 제출 시 카카오채널 팝업 차단 버그 수정

## 배경
신청폼(`/refer/trial`, `/refer/trial/[token]`)에서 "신청하기" 클릭 시 카카오채널
채팅창이 항상 뜨지 않는 문제 발견(김우석님 테스트 땐 뜨고, 김은정님 테스트 땐
안 뜸 — 타이밍에 따라 팝업이 차단되는 전형적 증상).

## 원인
7/13 티칭지 "본상담 예약하기" 버튼 작업 때 이미 겪었던 것과 동일한 문제로 추정.
`window.open()`을 신청서 저장(fetch) 요청이 끝난 뒤(then/await 이후)에 호출하고
있어서, 브라우저가 "사용자 클릭과 무관하게 나중에 열린 창"으로 판단해 팝업을
차단함. fetch 응답 속도에 따라 되기도/안 되기도 하는 것이 이 증상의 특징.

## 원칙 (7/13에 이미 확정된 패턴, 그대로 재적용)
`window.open()`은 클릭 핸들러 안에서 **가장 먼저, 동기적으로** 호출하고, 그 다음에
신청서 저장 등 비동기 작업을 진행할 것. `requestConsultCallback` /
`requestEventInquiryCallback` 구현 때와 동일한 순서.

## 수정
- `TrialApplicationForm.tsx`의 제출 핸들러 순서를 다음과 같이 변경:
  1. 클릭 즉시(동기) `window.open(NEXT_PUBLIC_KAKAO_CHANNEL_CHAT_URL)` 먼저 호출
  2. 그 다음 `await fetch(...)`로 신청서 저장 + 크레딧 적립 + 활동로그 등 처리
  3. 저장 실패 시에도 이미 열린 카카오 채팅창은 그대로 두고, 화면에는 에러만 표시
     (채팅창을 다시 닫지 않음 — 이미 원장님 쪽에는 문의가 들어온 상태이므로)
- 마감설문(`ExitSurveyForm.tsx`)도 동일한 패턴으로 구현되어 있는지 함께 점검
  (같은 문제가 있을 가능성 있음)
- 채널 추가 버튼(`KakaoChannelButton`)도 클릭 이벤트 핸들러 내부에서 직접
  동기적으로 호출되는 구조인지 확인 (SDK 함수 자체가 비동기 래핑되어 있지 않은지
  점검)

## 검증
- 새 시크릿 브라우저(카카오채널과 연결 이력 없는 상태)로 신청폼 제출 →
  카카오채팅창이 매번 확실히 뜨는지 반복 테스트(5회 이상)
- 네트워크를 일부러 느리게(개발자도구 쓰로틀링) 설정한 상태에서도 팝업이 뜨는지 확인
  (이게 원래 실패 재현 조건이므로 반드시 이 조건으로 확인할 것)
- 마감설문도 동일하게 반복 테스트
- 신청서 저장 자체가 실패하는 케이스(네트워크 끊김 등)에서도 채팅창은 정상적으로
  열려있는지 확인
- npx tsc --noEmit / npm run build 통과
