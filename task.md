### 오늘 세션 성격

### 오늘 결정된 사항

**1) 환자용 티칭지 (신규 개념, 설계만 완료)**
- 13-7(내부 스태프용 환자 통합뷰)과는 별개로, "환자에게 공개되는 URL 페이지"
  개념 신설. 카드뉴스 형태, 톡 메시지에 링크로 자연스럽게 삽입되어 발송,
  목적은 전환율/매출 증대
- 신규 모델 구상: PatientTeachingPage(token 랜덤고유값, patientId, contentJson
  스냅샷, viewCount, firstViewedAt 등)
- 공개 라우트 구상: /p/{token} (인증 없음, 한지·청자색 톤 카드뉴스)
- 생성 트리거: 담당자가 톡생성기에서 수동 버튼으로 생성 (자동생성 아님)
- 콘텐츠: 문구카드(AI 공감형) + 데이터카드(검사수치 변화 등) 혼합
- 접속 제한: 만료 없음으로 결정 (부하 문제 아니고 노출기간 리스크 문제로,
  현재 콘텐츠 성격상 리스크 낮다고 판단 — 추후 필요시 만료일 컬럼 확장 가능하게만
  설계)
- 이번 세션에서는 설계만, 구현은 다음 세션 이후

**2) 이벤트-톡 자연스러운 연계 아이디어 (설계 방향만 논의, 구현 시점 미정)**
- 기존 문제의식: 상시 이벤트 문자 발송에 반응이 거의 없어 피로도만 누적
- 새 방향: 이벤트를 별도로 살포하지 말고, 이미 나가는 톡(2일톡/7일톡/3회톡 등)
  이나 검사 시점에 자연스럽게 카드 하나로 얹어서 전달
- 예: 근력검사 재검에서 이전 대비 체중 유의미한 증가 발견 시, 그 사실을 개인화
  카드로 만들어 진행 중인 이벤트(예: 킬팻캡슐 3일체험)와 자연스럽게 연결
- 규림한의원 주요 프로그램(~10개) 전체를 한번에 설계하지 않고, 우선 1개
  파이프라인(킬팻캡슐)만 검증 후 확장하기로 함
- 조건 매칭은 단순 규칙(예: 검사 2회 이상 + 특정 수치 변화폭 이상)으로 시작,
  AI 추천 로직은 이후 단계로 미룸
- 자동발송 아님 — 담당자에게 "후보 제안"만, 최종 발송 판단은 사람이 유지
- 요구사항 문서에 신규 섹션(14번 등)으로 기록 필요, 구체 설계/구현은 다음 세션

**3) 검사(Examination) 기능 스키마 최종 확정 (실물 인바디/근감소증표 확인 후)**
- Patient 모델에 height(키) 고정값 필드 추가 결정
- 인바디(BodyCompositionRecord): 핵심 입력 3개(체중/체지방율/복부지방율=WHR)로
  단순화, VFA·부종지수는 제외
- 인바디에 사지근육량(오른팔/왼팔/오른다리/왼다리) 선택 입력 필드 추가 —
  입력 시 근력검사와 동일한 공용 SMI 계산 함수로 SMI 자동 산출
- BMI는 별도 저장 없이 체중+키로 화면에서 실시간 계산해 표시만
- 근력검사(StrengthTestRecord)는 기존 구조 유지, SMI 계산 로직을 인바디와 공유
- 검사종류는 확장 가능한 목록으로 관리하되, 계산식/판정 있는 검사(현재 2종)는
  코드로 관리, 향후 계산식 없는 단순 수치기록형(혈액검사 등) 필요 시 별도 범용
  테이블 추가 예정 (지금은 만들지 않음)
- 환자 목록(치료처방/내원체크)에서 "검사" 버튼으로 바로 입력 가능하도록 UX 변경
  (기존 독립페이지 재검색 흐름에서 전환)
