## 작업 지시서 — 검사(인바디/근력) 결과에 AI 해설 코멘트 추가

### 배경
- 검사 "환자와 함께보기" 화면(/patient-view/exam/[examType]/[id])이 현재 
  수치(체중/BMI/체지방율 등) + 판정 라벨("양호합니다") 정도만 보여줘서, 원장님이 
  환자 앞에서 직접 설명하실 때 화면 자체가 도와주는 게 별로 없음
- 실제 사례(강명화님 인바디, 체중 63.4kg/BMI 25.7/체지방율 42.4%/판정 양호)로 
  체감된 문제
- 해결 방향: 이 수치들을 바탕으로 AI가 **환자 친화적인 짧은 해설 코멘트**를 
  생성해 화면에 추가 (예: "체지방율 42.4%는 ○○ 정도로, ~~ 의미입니다" 같은 
  톤 — 위협적이지 않고 이해하기 쉽게)
- 프로그램티칭지 개발 과정에서 확립한 3원칙(① 실제 수치 인용 강제 ② 관련성 
  없으면 억지로 끼워넣지 않기 ③ 창작 금지, 원장 학술문구 범위 안에서만) 그대로 
  재사용

### 스키마 변경
- `BodyCompositionRecord`, `StrengthTestRecord`에 `aiExplanation`(String, 
  nullable) 필드 추가
- 마이그레이션 적용, 기존 레코드는 null로 유지 (소급 생성 안 함 — 아래 
  "기존 레코드 처리" 참고)

### 생성 로직
- 신규 함수 (예: `src/lib/exam-explanation.ts`) — 기존 OpenAI 엔진(gpt-4o-mini) 
  재사용
- 입력재료: 검사종류(인바디/근력), 실제 측정값 전체, 계산된 판정 라벨, 
  (있다면) 직전 검사 대비 변화량
- 프롬프트 원칙:
  1. **실제 수치 인용 강제** — "정상 범위는 ~~입니다" 같은 일반론 금지, 
     이 환자의 실제 숫자를 반드시 언급
  2. **의학적 진단/효능 창작 금지** — 질병명 언급, 치료 효과 단정 금지, 
     "~일 수 있습니다" 정도의 신중한 어조 유지
  3. **환자 친화적 어조** — 전문용어(BMI, SMI 등)는 풀어서 설명, 짧게 
     2~4문장 이내
  4. **판정이 낮은 경우에도 불안 조성 금지** — 있는 그대로 설명하되, 
     다음 단계(재검/생활습관 등)로 자연스럽게 이어지는 긍정적 톤 유지
  5. 자체검토 항목: 실제 수치 인용 여부 / 진단성 표현 여부 / 어조 적절성
- max_tokens는 짧은 코멘트이므로 200 내외로 설정

### 생성 시점
- 검사 등록/저장 시점에 **동기적으로 함께 생성** (원장님이 저장 직후 바로 
  "환자와 함께보기"로 넘어가는 실사용 흐름이므로, 클릭 시점에 지연 없이 
  바로 보여야 함)
- AI 생성 실패 시에도 **검사 저장 자체는 반드시 성공**해야 함 (7/10 세션에서 
  전수 수정했던 "저장 무음 실패 방지" 원칙 유지) — 실패 시 `aiExplanation`은 
  null로 남기고 에러는 서버 로그로만 기록, 사용자에게 저장 실패로 보이지 않게

### 표시 위치
- `/patient-view/exam/[examType]/[id]` — 수치/판정 아래에 해설 코멘트 문단 추가
- `/patient-view/exam-report/[patientId]` (종합 리포트, 2회 이상 검사 시) — 
  가장 최근 검사의 해설 코멘트 노출

### 기존 레코드 처리
- 이미 저장된 과거 검사 기록(aiExplanation=null)을 "환자와 함께보기"로 열었을 때: 
  화면에서 즉석으로 생성 요청 (버튼 클릭 또는 자동 1회 생성 후 캐싱) — 
  매번 새로 만들지 않고 한 번 생성되면 저장해서 재사용

### 검증 체크리스트

- [ ] 신규 근력검사도 동일하게 확인
- [ ] 실제 수치가 코멘트 안에 그대로 인용되는지 확인 (뭉뚱그린 일반론 아닌지)
- [ ] 판정이 낮은 케이스(예: 약함)에서도 불안 조성 없이 자연스러운 톤인지 확인
- [ ] AI 생성 실패를 강제로 재현했을 때 검사 저장 자체는 정상 완료되는지 확인
- [ ] 기존(과거) 검사 기록 열람 시 즉석 생성 및 캐싱 동작 확인
