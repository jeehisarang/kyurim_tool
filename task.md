# 작업지시서 — HRV "환자와 함께보기" 화면 가독성 개선

## 1. 배경

- 대상 화면: HRV 검사결과 환자용 화면 (/patient-view/exam/hrv/[id] 등 —
  정확한 경로/컴포넌트는 코드에서 확인해서 특정할 것. 원장 확인화면
  /examinations/hrv/[id]은 이번 작업 대상 아님, 절대 건드리지 말 것)
- 원장님 피드백: 현재 화면(기기판독요약/임상적의미/생활관리안내/한의학적해석/
  안전안내 5개 카드 구조)이 전체적으로 텍스트 밀도가 높고 평면적이라
  환자가 한눈에 핵심을 파악하기 어려움. 아래 3가지 개선 필요:
  1. 수치(혈관건강지수, 평균맥박, 스트레스지수, 혈관건강도 등급 등) —
     정상/비정상 여부에 따라 색상 구분(정상=녹색 계열, 비정상=빨강 계열) +
     글자 크기 강조
  2. 중요 문구(핵심 판정 내용) — 굵게 + 확대
  3. 전반적 가독성(폰트 크기, 줄간격, 여백) 개선

## 2. 착수 전 확인 필요 (설계 먼저 제시)

구현 전에 아래 사항부터 코드/데이터 확인 후 계획만 정리해서 원장님께
먼저 보고할 것 (승인 후 실제 구현 진행):

1. 수치가 구조화된 필드로 DB에 저장돼 있는지 확인
   - HrvTestRecord 스키마에 혈관건강지수/평균맥박/스트레스지수/혈관건강도
     등급이 각각 별도 컬럼(숫자/문자열)으로 있는지, 아니면 "기기 판독 요약"
     문단 전체가 AI가 생성한 자유 텍스트 한 덩어리로만 저장되는지 확인
   - 구조화된 필드가 있는 경우: "기기 판독 요약" 섹션을 문장형에서
     라벨+수치 칩/뱃지 형태(예: "혈관건강지수 -11"을 별도 UI 요소로)로
     일부 재구성하는 방향을 제안하고, 그 위에서 정상/비정상 색상을 입힐 것
   - 구조화된 필드가 없고 AI 생성 텍스트에 수치가 섞여 있는 경우: 정규식
     등으로 텍스트에서 파싱해 색칠하는 방식은 깨지기 쉬우므로 권장하지 않음
     — 이 경우 원장님께 "구조화 저장으로 먼저 바꿀지, 아니면 색상 강조는
     포기하고 굵게/크게 정도만 적용할지" 선택지를 제시하고 답변 받은 뒤 진행

2. 정상/비정상 판정 기준 확인
   - 혈관건강지수, 평균맥박, 스트레스지수, 혈관건강도 등급 각각의 정상 범위
     기준이 기존 코드(exam-thresholds.ts 유사 파일)에 이미 있는지 확인
   - 없다면 임의로 만들지 말고, 원장님께 각 지표별 정상 범위를 물어본 뒤
     상수 파일로 관리할 것 (다른 검사와 동일 패턴)

3. "중요 문구"의 판단 기준 제안
   - 무엇을 "중요 문구"로 볼지 애매하므로, 다음 후보를 원장님께 제안해서
     확정받을 것:
     - (안 A) 각 섹션의 핵심 결론 문장 1개씩만 굵게/확대
     - (안 B) 패턴명(간기울결, 심비양허 등 한의학적 해석의 병기명)만 굵게/확대
     - (안 C) A+B 둘 다
   - 임의 판단으로 구현하지 말고 반드시 확정 후 진행

## 3. 동작 규칙 (설계 승인 후 적용)

- 정상 수치: 녹색 계열(색상코드 명시할 것, 기존 디자인 토큰 있으면 그것 사용),
  비정상 수치: 빨강/경고 계열 — 색상 대비(contrast)는 텍스트 가독성 기준
  충족할 것 (밝은 배경에 옅은 색 금지)
- 강조 대상 수치는 폰트 크기도 주변 텍스트보다 한 단계 이상 키울 것
- 중요 문구는 font-weight: bold + 폰트 크기 확대, 단 문단 전체를 굵게
  만들지 말고 확정된 기준(2-3안)에 따라 문장/구절 단위로만 적용
- 전체 가독성: 기본 폰트 크기 상향, 줄간격(line-height) 여유있게, 카드 간
  여백도 재검토 — 단 안전 안내 카드의 기존 경고색 배경 스타일은 유지
- 원장 확인화면(/examinations/hrv/[id])에는 이번 변경사항이 전파되지
  않도록 컴포넌트 분리 여부 확인 후 작업 (공유 컴포넌트라면 환자용
  props/variant로만 분기)

## 4. 검증 체크리스트

- 정상 수치 케이스 1건 + 비정상 수치 케이스 1건(예: 전태균님 기록) 모두
  실기기 브라우저로 확인, 색상이 의도대로 뒤바뀌지 않는지 확인
- 팝업창(1100×980) 크기에서 잘리거나 스크롤 깨지는 부분 없는지 확인
- 원장 확인화면은 기존 그대로인지 별도로 확인 (이번 작업으로 영향 없어야 함)
- 실데이터 화면에서 자동화 스크립트 실행 금지 — 검증은 테스트 전용 환자
  (예: 426번)로만 진행할 것

## 5. 완료보고 형식

- 1번(구조화 여부), 2번(정상범위 기준), 3번(중요문구 기준) 각각에 대해
  코드 확인 결과 + 제안안을 먼저 보고 (구현 전 승인 요청)
- 승인 후 구현 완료 시: 수정 파일 목록, 스크린샷 또는 실확인 결과, git
  commit 해시
